1. Executive summary
This supersedes the earlier greenfield concept. After reviewing the actual codebase, the reality is: tradiee-mobile is already a serious, offline-first field-service app, and the web app (tradiee-app) is far ahead of it — it already has a unified inbox, two-way SMS threads, enquiries, bookings, a customer portal and a channel-aware notification layer. The mobile app has none of that yet, by design: every Growth-Engine sprint explicitly says “do not touch the mobile app.” So this is not a rebuild — it is an enhancement and parity job.
Two things drive the overhaul, and both are exactly what you asked for. (1) A mobile Unified Inbox that mirrors the web /messages page — enquiries, website leads, booking requests and SMS threads in one place, answerable in one or two taps, with real-time push. (2) One-touch “On my way” with an auto-calculated ETA — a mobile-native feature that finally uses the GPS, LocationIQ geocoding, site coordinates and geofencing the app already ships.
The guiding rule throughout: keep the architecture and reuse the web APIs. The phone should consume /api/messages/*, /api/sms/send and /api/notify rather than re-implement business logic on-device, and it must honour the same channel-aware, Twilio-dark-safe contract as the web — SMS paths are built now but stay cleanly disabled until Twilio is enabled.
Delivery: this document is the plan (opusplan-style — review before build). It is paired with an updated interactive prototype (IndustryForms_Mobile_Prototype.html) in the app&apos;s real orange brand, showing the Inbox, enquiry reply, one-touch ETA and the redesigned job screen. On sign-off, section 13 is a file-by-file build plan and section 14 phases the work to slot in beside the web sprints.

5. Design principles &amp; tokens
The app already has a consistent visual language (orange #F97316 on light grey). We formalise it into tokens and hold every screen to seven principles.
1.	One-thumb, on-site. Primary actions in the bottom thumb zone; ≥48dp targets; nothing critical near the top edge.
2.	Speed beats features. Every hero action ≤2 taps; defaults and templates over typing.
3.	Reuse the server, don&apos;t duplicate it. The phone calls the web APIs; business rules live in one place.
4.	Channel-aware &amp; Twilio-dark-safe. SMS UI is built but disabled until Twilio is live; nothing SMS-related may hard-fail while dark.
5.	Offline-tolerant. Reads work offline via PowerSync; sends queue and reconcile — matching the existing timer/geofence patterns.
6.	Legible outdoors. 16–17px base, bold labels, high contrast; support Dynamic Type.
7.	Branded automatically. Company name and tech name merge into every outbound message; no hand-typed signatures.

7. Hero 1 — Mobile Unified Inbox
Mirror the web /messages experience on the phone: one list of every inbound communication, answerable instantly. It consumes the web&apos;s existing endpoints so the two stay in lock-step.
Data &amp; endpoints (reuse — do not rebuild)
GET /api/messages/conversations — unified feed (SMS threads + enquiries + web leads + bookings), already normalised server-side into ConversationSummary.
GET /api/messages/thread — a selected thread; POST /api/messages/action — mark read/closed/spam, create/link customer, create quote/job.
POST /api/sms/send — outbound reply, dark-aware: render a disabled “SMS not enabled — configure Twilio in Settings” state until live (same contract as the web thread component).
Add enquiries and customer_messages to lib/powersync/schema.ts + sync-rules.yaml so the inbox reads offline; writes (reply, triage) go through the APIs online and reconcile.
Real-time push (upgrade the poll)
The web should call /api/notify → Expo push (token already on profiles.expo_push_token) on new inbound. Register notification categories/actions (Reply · Quote · Call) so a lead can be answered from the lock screen.
Keep the Notifications activity feed as history; the Inbox tab becomes the live surface with an unread badge.
Screen — Inbox
Purpose
Answer “who is waiting on me?” — every enquiry, lead, booking and SMS in one list.
Elements
Tabs mirroring web (Open / Unread / Enquiries / Bookings / Unmatched / Closed); rows with avatar, name (or sender for unmatched), snippet, channel icon, status pill, unread dot; pull-to-refresh + 15s poll.

Convert-to-quote/job pre-fills the customer from the enquiry.

8. Hero 2 — One-touch “On my way” + ETA
A mobile-native feature that finally uses the location stack already in the app. From a job, a thread, or the schedule, one tap sends the customer a branded, ETA-accurate arrival message — no typing.
How it works with what exists
1.	Read current GPS via expo-location (already permissioned for background tracking) and the job&apos;s site lat/lng from customer_sites.
2.	Compute a live drive-time ETA (LocationIQ Directions — the key is already configured as EXPO_PUBLIC_LOCATIONIQ_KEY; or a small /api/eta route if you prefer to keep the key server-side).
3.	Send a branded message through the channel-aware layer — ideally a new /api/sms/eta (sibling of the existing /api/sms/quote and /api/sms/invoice) that logs to customer_messages and uses notify() so it emails now and SMSes when Twilio flips.
4.	Optionally attach a live tracking link built on the existing portal/[token] surface; it auto-expires on arrival.
5.	Arrival is already detected — the geofence auto-check-in in lib/location/tracking.ts fires within 150 m. Hook it to flip the ETA state to “Arrived” and (optionally) auto-send an arrival message.
Screen — On my way
Purpose
Tell the customer you&apos;re coming, accurately and branded, in one tap.
Elements
Big ETA number + distance; route map; branded message preview (merge fields resolved); alternate chips (Running late / Arrived / Reschedule); dark-aware send.
Primary action
Send (1 tap) — with optional live-tracking link.
Acceptance criteria
From the job screen, “On my way” sends a correctly-branded, ETA-accurate message in one tap (email now; SMS when Twilio live).
Arrival flips state automatically via the existing geofence; the event is logged on the job/thread.
With Twilio dark, ETA still sends via the live channel (email/portal) and never hard-fails.

Keep every capability in app/jobs/[id].tsx (timer, materials, photos, notes, signature sign-off, complete-&amp;-invoice, Tap-to-Pay). Change the layout so the on-site actions are reachable one-handed.
Add a horizontal quick-action row under the header: On my way · Navigate · Call · Message · Start timer · Add photo · Forms. (Call/Navigate exist but are buried in meta rows; Message/On-my-way/Forms are new.)

10. Architecture &amp; integration rules (must-follow)
These keep the mobile changes consistent with the web app and its sprint contract. Breaking them is how the two apps drift apart.
Reuse web APIs; don&apos;t duplicate logic on device. Inbox, SMS, ETA and invoicing all call existing routes.
Channel-aware, Twilio-dark-safe. Every send goes through the notify/SMS layer that no-ops cleanly while dark; SMS UI shows the disabled state.
Session client, never the service client, from the phone. All calls run under the user&apos;s Supabase session (RLS-scoped), matching the web rule.
Idempotent writes. Reuse the timer/geofence reconciliation pattern already in the codebase for any queued mobile write.
PowerSync additions are additive. Add tables to schema.ts and sync-rules.yaml together; respect the mig-047 columns (read_at, status, source).
Respect roles. Inbox/messaging is owner/admin on web; mirror that gating on mobile (staff already hidden from Quotes).

16. Risks &amp; open decisions
Twilio is dark. ETA and replies must send by email/portal now and flip to SMS on go-live with no code change — reuse the web&apos;s notify() contract.
Background location battery. ETA reuses existing tracking; don&apos;t add a second location subscription — piggyback the running task.
Offline writes. Replies/triage are online-only v1 (queued with clear pending state); don&apos;t attempt offline SMS.
iOS notification actions. Lock-screen Reply needs a notification category + a background handler; scope in M1.
Decide: ETA routing via LocationIQ on-device vs a server /api/eta (keeps the key off the client). Recommend server route.
Decide: live-tracking link scope/expiry on the portal (privacy — expire on arrival).
